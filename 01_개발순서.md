# 01_개발순서.md

## 문서 목적

이 문서는 LunchLab ERP 프로젝트를 이어받는 개발자 또는 다른 AI 모델이
현재 프로젝트의 전체 개발 순서와 선후관계를 오해 없이 이해할 수 있도록 정리한 문서다.

이 문서는 “무엇을 먼저 읽고, 무엇을 먼저 수정하고, 어떤 순서로 다음 단계에 들어가는가”를 정의한다.
세부 구조 기준은 `02_개발문서.md`,
실행 절차는 `03_개발가이드문서.md`,
검증 기준은 `04_전체 구현 체크리스트 및 코드기준안.md`,
공식 변경 이력은 `05_불일치 정리 및 통합기준 제안.md`,
현재 실작업 범위는 `06_Phase 3. 식단 관리 프로세스.md`,
인수인계 기준은 `07_HANDOFF.md`를 따른다.

---

## 1. 현재 프로젝트의 실제 위치

현재 프로젝트의 실제 상태는 다음과 같다.

- Sprint 1은 완료되었다.
- Sprint 2는 진행 중이다.
- Sprint 2의 완료 지점은 `Phase 5 — 식단 그룹 UI`이다.
- Sprint 2의 기존 미완료 작업은 아래와 같다.
  - Phase 2-e — 부자재-공급업체 연결 UX
  - Phase 6 — 식단 캘린더 뷰
  - Phase 7 — 슬롯 상세 에디터
  - Phase 8 — MealCount + MealPlanAccessory 서비스/UI
  - Phase 9 — 소요량 자동 산출 서비스
  - Phase 10 — 테스트 작성
  - Phase 11 — 페이지 통합 + Sprint 2 QA
- 다만 MealPlan 도메인의 구조 문제로 인해, 위 미완료 작업을 그대로 진행하지 않고,
  Sprint 2 내부에 **구조 재정의 보강 작업**을 삽입한 뒤 다시 Sprint 2를 재개한다.

이 문서의 핵심 원칙은 다음과 같다.

> 기존 계획을 삭제하지 않는다.  
> 기존 완료 이력을 요약해서 날리지 않는다.  
> 구조 재정의는 기존 계획을 대체하지 않고, 그 사이에 추가되는 보강 작업으로 처리한다.

## 2. 가장 먼저 읽을 문서 순서

프로젝트를 처음 이어받는 경우 아래 순서로 읽는다.

1. `PROGRESS.md`
2. `01_개발순서.md`
3. `02_개발문서.md`
4. `03_개발가이드문서.md`
5. `04_전체 구현 체크리스트 및 코드기준안.md`
6. `05_불일치 정리 및 통합기준 제안.md`
7. `06_Phase 3. 식단 관리 프로세스.md`
8. `07_HANDOFF.md`

이 순서를 권장하는 이유는 다음과 같다.

- `PROGRESS.md`는 전체 히스토리와 현재 단계가 어디인지 알려준다.
- `02_개발문서.md`는 구조와 규칙의 기준선을 알려준다.
- `03_개발가이드문서.md`는 실제 작업 절차를 알려준다.
- `04_체크리스트`는 완료 판정 기준을 알려준다.
- `05_불일치 정리`는 왜 구조가 바뀌었는지를 설명한다.
- `06_Phase 3 문서`는 지금 당장 해야 하는 작업 범위를 보여준다.
- `07_HANDOFF.md`는 다음 시작점과 금지 추론 항목을 정리한다.

## 3. 현재 실행 순서의 큰 그림

현재부터의 실제 개발 순서는 아래와 같다.

### 3.1. 기존 완료 이력 보존
가장 먼저 해야 하는 일은 “이미 끝난 작업을 다시 줄여 쓰거나 대체하지 않는 것”이다.

반드시 보존해야 하는 항목:
- Sprint 1 전체 완료 이력
- Sprint 2 Phase 1, 2, 2-b, 2-c, 2-d, 3, 4, 5 완료 이력
- Sprint 2 미완료 Phase
- Sprint 3~8 전체 계획
- 커밋, 변경 파일, 이슈 해결 흐름

### 3.2. 구조 재정의 기준 확정
그 다음 MealPlan 구조 재정의 기준을 문서에 먼저 확정한다.

확정 내용:
- MealPlanGroup = 날짜 그룹
- MealPlan = 식사타입 × lineup
- MealPlanSlot = 실행 배정 단위
- MealCount = 상태가 아닌 데이터
- MaterialRequirement / CookingPlan / 자동생성 기준 입력 = MealPlanSlot

### 3.3. schema 및 migration 수정
문서 기준이 확정되면 `schema.prisma`를 수정하고 migration을 만든다.

### 3.4. seed / schema / service / action 수정
그 다음 시드와 meal-plan 관련 계층을 새 구조로 정렬한다.

### 3.5. UI 재구성
그 다음 `/meal-plans/page.tsx`와 보조 컴포넌트를 재구성한다.

### 3.6. 자동생성 연결 기준 정리
그 다음 MealCount, MealPlanAccessory, MaterialRequirement, CookingPlan 연결 기준을 정리한다.

### 3.7. 테스트 보강
그 다음 구조 재정의에 맞춰 테스트를 보강한다.

### 3.8. Sprint 2 재개
그 다음 원래 Sprint 2의 미완료 작업으로 복귀한다.

재개 순서:
1. Phase 2-e
2. Phase 6
3. Phase 7
4. Phase 8
5. Phase 9
6. Phase 10
7. Phase 11

## 4. 현재 라운드에서 절대 바꾸면 안 되는 선후관계

다음 순서는 절대 뒤집지 않는다.

1. 문서 기준 확정
2. schema 수정
3. migration 검증
4. seed 수정
5. service 수정
6. action 수정
7. UI 수정
8. 자동생성 기준 정리
9. 테스트 보강
10. 문서 갱신
11. Sprint 2 재개

특히 아래는 금지한다.

- schema를 안 바꾼 상태에서 UI부터 대폭 변경
- MealCount 먼저 구현
- MaterialRequirement 먼저 구현
- CookingPlan 먼저 구현
- 구조 재정의 전에 Phase 6~11 강행
- 기존 Sprint 이력을 “정리한다”는 이유로 줄여 적기

## 8. 현재 라운드 완료 기준

현재 구조 재정의 라운드는 아래 조건을 모두 만족해야 완료다.

- 문서 간 구조 정의가 일치한다
- schema와 migration이 적용된다
- seed가 재실행 가능하다
- MealPlanGroup / MealPlan / MealPlanSlot이 새 구조를 반영한다
- MealCount는 상태와 분리된다
- UI가 날짜 그룹 → 식사타입 × lineup → 슬롯 배정 구조를 제공한다
- 자동생성 연결 기준이 문서화된다
- 테스트가 PASS한다
- `PROGRESS.md`에 반영된다
- 이후 Sprint 2 미완료 Phase를 다시 시작할 수 있다
