# 02_개발문서.md

## 문서 목적

이 문서는 LunchLab ERP 프로젝트의 구조 설계, 계층 규칙, 도메인 책임, 상태 전이, 연결 기준, 금지 규칙을 정의하는 핵심 개발 문서다.

이 문서는 “어떤 구조가 맞는가”를 정의한다.
실행 순서는 `01_개발순서.md`,
실행 절차는 `03_개발가이드문서.md`,
검증 기준은 `04_전체 구현 체크리스트 및 코드기준안.md`,
공식 변경 이력은 `05_불일치 정리 및 통합기준 제안.md`를 따른다.

## 1. 시스템 전반 구조

프로젝트는 Next.js App Router 기반 monolithic full-stack 구조를 사용한다.

기본 흐름은 다음과 같다.

UI  
→ Server Action / API Route  
→ Zod Schema Validation  
→ Session / Permission / Scope Check  
→ Service Layer  
→ Prisma Client  
→ Database

핵심 원칙은 다음과 같다.

- 클라이언트 컴포넌트는 DB에 직접 접근하지 않는다
- 도메인 규칙은 service 계층이 가진다
- action은 오케스트레이션 계층이다
- schema(zod)는 입력 검증과 타입 정규화를 담당한다
- transaction이 필요한 다중 쓰기는 service에서 처리한다
- 권한과 감사 로그는 쓰기 작업에서 기본값이다

## 2. 시스템 도메인 기준

### 2.1 Company / Location / ProductionLine 역할 구분
- **Company**: 계획과 기준 정보의 소유 단위
- **Location**: 재고 보유 및 물류 단위
- **ProductionLine**: 실제 조리/생산 실행 단위

### 2.2 BOM과 Inventory 분리
- BOM은 기준 소요량이다
- Inventory는 실제 재고 흐름이다
- 실제 원가는 BOM이 아니라 실제 소비 데이터로 계산한다

### 2.3 Append-only 원칙
아래 영역은 append-only 성격을 유지한다.
- inventory transaction
- consumption
- audit log
- month-end correction 관련 로그

### 2.4 Semi-product 규칙
- 레시피 BOM은 semi-product를 참조할 수 있다
- semi-product BOM은 material만 참조한다
- semi-product의 중첩 참조는 허용하지 않는다

### 2.5 Month-end lock
- 마감 이후 과거 재고/원가를 직접 수정하지 않는다
- 필요한 경우 adjustment / reversal 방식으로 처리한다

## 3. 공통 계층 규칙

### 3.1 Schema 계층
책임:
- 입력 검증
- 숫자/문자열/날짜 제약
- enum coercion
- 기본값 처리
- UI 입력과 도메인 입력 사이 타입 정렬

규칙:
- action에 raw form data를 그대로 넘기지 않는다
- parsing 실패 메시지는 UI가 처리 가능한 형식으로 반환한다

### 3.2 Action 계층
책임:
- session 확보
- permission 검사
- schema parse
- service 호출
- audit log 기록
- error mapping
- revalidatePath 또는 재조회 트리거

규칙:
- action에서 Prisma 직접 조작을 최소화한다
- action은 “도메인 판단”보다 “흐름 제어”에 집중한다
- `handleActionError` 패턴을 유지한다

### 3.3 Service 계층
책임:
- 비즈니스 로직
- 상태 전이 검증
- relation include 정책
- deep copy / cascade delete
- transaction 처리
- soft-delete where 조건 관리

규칙:
- service는 테스트 가능해야 한다
- NOT_FOUND / DUPLICATE / DOMAIN_ERROR는 service에서 명시적으로 다룬다

### 3.4 UI 계층
책임:
- 목록/상세 렌더링
- dialog / confirm / badge / form UX
- toast 메시지
- loading / empty / error 처리

규칙:
- UI가 도메인 규칙을 추론하지 않는다
- action 결과를 기준으로 동작한다
- console.log를 사용하지 않는다

## 6. MealPlan 영역 구조 재정의

### 6.1 왜 재정의가 필요한가
현재 기본 구현은 CRUD 자체는 되지만, 아래 문제가 있다.

1. `MealPlanGroup`가 날짜 그룹과 lineup 책임을 동시에 가진다
2. `MealPlan`이 식사타입 × lineup 구조를 안정적으로 표현하기 어렵다
3. `MealPlanSlot`이 실제 실행 배정 정보가 부족하다
4. 이 상태에서는 MealCount, MaterialRequirement, CookingPlan, 자동생성 연결이 불안정하다

즉, “기능이 없음”이 아니라,
**후속 Sprint를 안정적으로 지탱하지 못하는 구조**가 문제다.

---

## 6.2 재정의 후 공식 구조

### MealPlanGroup
역할:
- 날짜 중심 상위 그룹
- 하루 단위 컨테이너
- 상태 전이의 중심
- 상위 메모/관리 정보 소유

가지면 안 되는 책임:
- lineup의 직접 소유
- 실행 배정의 직접 소유
- 실제 소요량 계산의 직접 소유

### MealPlan
역할:
- MealPlanGroup 하위 엔티티
- 식사타입 × lineup 조합
- 템플릿 기반 식단의 단위
- 여러 Slot을 가진다

예시:
- 2026-05-21 / 중식 / 일반 라인업
- 2026-05-21 / 중식 / 키즈 라인업

### MealPlanSlot
역할:
- 실제 실행 배정 단위
- recipe / BOM / semi-product / supplier item / productionLine / quantity / note 등 수용
- MaterialRequirement, CookingPlan, 자동생성의 기준 입력

추가 또는 보강되어야 할 정보 후보:
- `supplierItemId`
- `productionLineId`
- `itemName`
- `note`
- `sortOrder`
- `quantity`
- 필요 시 슬롯 타입별 보조 필드

### MealCount
역할:
- 수량 입력 데이터
- estimatedCount / finalCount 등 운영값
- 상태가 아니다

금지:
- MealCount로 상태를 표현하는 것
- MealCount 입력 시 상태가 자동 변경되는 것

## 7. MealPlan 도메인 상태 규칙

### 7.1 상태 전이 주체
상태 전이의 주체는 MealPlanGroup이다.

### 7.2 허용 상태
- `DRAFT`
- `CONFIRMED`
- `IN_PROGRESS`
- `COMPLETED`
- `CANCELLED`

### 7.3 대표 규칙
- DRAFT → CONFIRMED 가능
- CONFIRMED → IN_PROGRESS 가능
- IN_PROGRESS → COMPLETED 가능
- COMPLETED 이후 수정 제한
- CANCELLED 이후 복구는 별도 정책 필요
- MealCount는 상태와 분리

---

## 8. MealPlan 도메인 동작 규칙

### 8.1 생성
- Group 생성: 날짜 기준 상위 그룹 생성
- Plan 생성: Group 하위에 식사타입 × lineup 생성
- Slot 생성: Plan 하위에 실행 배정 단위 생성

### 8.2 복사
- Group 복사 시 하위 Plan / Slot 구조를 deep clone
- 복제 후 상태는 보수적으로 DRAFT
- count와 실행 결과 데이터는 정책적으로 분리

### 8.3 삭제
- Group 삭제는 하위 구조를 고려한 cascade 정책 필요
- soft-delete 또는 명시적 삭제 정책을 문서와 code에서 일치시켜야 함
- transaction 사용

### 8.4 자동생성
- 자동생성의 기준 입력은 MealPlanSlot
- MealCount, accessory, slot assignment가 연결되어 MaterialRequirement / CookingPlan / 발주 초안으로 확장
- 재실행 정책은 멱등성을 가져야 함

## 11. 후속 연결 기준

## 12. 금지 규칙

다음은 현재 구조 재정의 단계에서 금지한다.

- Group에 lineup 책임을 다시 고정하는 것
- MealCount를 status처럼 다루는 것
- Slot 구조가 정리되기 전에 MaterialRequirement를 본 구현하는 것
- CookingPlan을 구조 재정의 전에 먼저 진행하는 것
- UI 구조만 먼저 바꾸고 schema를 나중에 맞추는 것
- 문서 기준 없이 임의 필드 추가
- soft-delete / audit / permission / transaction 패턴 누락

---

## 13. 현재 구조 재정의 완료 기준

구조 재정의는 아래를 모두 만족해야 완료다.

- 문서 정의와 schema 정의가 일치한다
- seed가 새 구조를 반영한다
- service/action 시그니처가 새 구조를 반영한다
- UI가 날짜 그룹 → 식사타입 × lineup → 슬롯 배정 구조를 제공한다
- count와 상태가 분리된다
- 자동생성 연결 기준이 문서화된다
- 테스트가 PASS한다
- Sprint 2 원래 미완료 Phase를 이어갈 수 있다
