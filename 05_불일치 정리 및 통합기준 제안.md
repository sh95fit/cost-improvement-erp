## 2. 기존까지의 주요 구조 변경 이력

### 2.1 Sprint 1 계열 변경

#### 단위 관리 중앙화
- 변경 전: 단위 자유입력 및 화면별 중복 관리 요소 존재
- 변경 후: UnitMaster 중심 DB 관리 체계로 통일
- 영향:
  - material
  - subsidiary
  - unit conversion
  - `/units` 페이지
- 사유:
  - 자유입력은 데이터 품질을 떨어뜨리고 재사용성과 검증 가능성을 해쳤음

#### Container 삭제 의존성 검증
- 변경 전: 참조 관계가 있어도 삭제 시도 가능
- 변경 후: MealTemplate / RecipeBOMSlot 참조를 사전 검증
- 영향:
  - container service/action/UI
- 사유:
  - 잘못된 삭제로 후속 구조 붕괴 가능

#### BOM 복제 + 편집 UX
- 변경 전: ACTIVE BOM 이후 수정 흐름이 막힘
- 변경 후: 복제 → DRAFT 편집 → ACTIVE 확정 흐름 정립
- 사유:
  - 운영상 버전 관리 필요

#### v5 Prisma 구조 전환

## 3. 현재 핵심 불일치

현재 가장 큰 불일치는 MealPlan 영역이다.

### 불일치 1 — MealPlanGroup 책임 과다
- 현재 기본 구현에서는 Group이 날짜 그룹 이상의 책임을 가지고 있음
- 결과:
  - lineup 구조가 상위에 고정됨
  - 같은 날짜에 여러 lineup을 유연하게 표현하기 어려움

### 불일치 2 — MealPlan 책임 불안정
- 현재 MealPlan은 식사타입 × lineup 구조를 공식적으로 안정화하지 못한 상태
- 결과:
  - UI 카드 구조가 장기적으로 흔들릴 수 있음
  - 후속 count / requirement / cooking 연결이 불명확해짐

### 불일치 3 — MealPlanSlot 정보 부족
- Slot은 실행 배정 단위가 되어야 하는데 필요한 필드가 부족
- 결과:
  - supplier item, production line, 실행 단위 메모 등을 담기 어려움
  - MaterialRequirement / CookingPlan 연결 기준이 모호함

### 불일치 4 — MealCount의 역할 혼동 가능성
- 기존 문서 일부 흐름에서는 count와 상태가 혼동될 여지가 있음
- 결과:
  - count 입력 시 상태를 바꾸는 잘못된 구현이 생길 수 있음

## 4. 공식 통합 기준

이 문서에서 아래 기준을 공식 기준으로 확정한다.

### 기준 1 — MealPlanGroup
- 날짜 중심 상위 그룹이다
- 하루 단위 컨테이너다
- 상태 전이의 기준 단위다
- lineup 책임을 직접 가지지 않는다

### 기준 2 — MealPlan
- MealPlanGroup 하위 엔티티다
- 식사타입 × lineup 조합이다
- 여러 Slot을 가진다

### 기준 3 — MealPlanSlot
- 실제 실행 배정 단위다
- recipe/BOM/semi-product/supplier item/production line/quantity/note 등의 입력 기준이다
- MaterialRequirement / CookingPlan / 자동생성의 기준 입력이다

### 기준 4 — MealCount
- 상태가 아니다
- 운영 입력 데이터다
- estimated/final count 등의 필드로 관리한다

### 기준 5 — 자동생성 기준
- 자동생성은 MealPlanSlot을 기준 입력으로 정렬한다
- MealCount와 accessory는 보조 입력이다
- 멱등성 정책을 반드시 가진다

## 6. 변경 방식 원칙

이번 구조 재정의는 아래 방식으로 반영한다.

- 기존 Sprint 1~8 계획 삭제 금지
- 기존 Sprint 2 완료 이력 삭제 금지
- 기존 Sprint 2 미완료 Phase 삭제 금지

- 문서 간 연결 정보는 기존 내용을 대체하지 않고 참조 정보로 추가

## 7. 현재 시점의 공식 결론

현재 시점에서 공식 결론은 아래와 같다.

1. Sprint 2는 아직 끝나지 않았다
2. Sprint 2의 원래 계획은 유지된다
3. 다만 Sprint 2 Phase 5까지 구현한 MealPlan 구조는 후속 작업을 안정적으로 받기 어렵다
4. 따라서 Sprint 2 내부에 구조 재정의 보강 작업을 선행 삽입한다
5. 구조 재정의 완료 후 Sprint 2의 원래 미완료 Phase를 다시 진행한다
