# 03_개발가이드문서.md

## 문서 목적

이 문서는 현재 프로젝트에서 가장 중요한 작업인
“MealPlan 도메인 구조 재정의 보강”을 실제로 어떻게 수행할지를 단계별로 설명하는 실행 절차 문서다.

이 문서는 현재 라운드에 집중한다.
전체 순서는 `01_개발순서.md`,
설계 기준은 `02_개발문서.md`,
검증 기준은 `04_전체 구현 체크리스트 및 코드기준안.md`,
변경 이유는 `05_불일치 정리 및 통합기준 제안.md`,
현재 작업 현황은 `06_Phase 3. 식단 관리 프로세스.md`,
handoff 기준은 `07_HANDOFF.md`를 따른다.

## 2. Step 1 — 구조 기준 문서 확정

### 목적
코드보다 먼저 구조 기준을 고정한다.

### 해야 할 일
- `PROGRESS.md`에 현재 상태 / 완료 범위 / 남은 작업 / 보류 범위 / handoff 기준을 보강한다
- `02_개발문서.md`에 MealPlan 구조 재정의 기준을 명시한다
- `05_불일치 정리 및 통합기준 제안.md`에 변경 이유를 기록한다
- `06_Phase 3. 식단 관리 프로세스.md`에 현재 구조와 실제 작업 단계를 반영한다
- `07_HANDOFF.md`에 다음 작업자용 기준선과 금지 추론 항목을 정리한다

### 이 단계에서 확정해야 하는 것
- MealPlanGroup = 날짜 그룹
- MealPlan = 식사타입 × lineup
- MealPlanSlot = 실행 배정 단위
- MealCount = 상태가 아닌 데이터
- 자동생성 입력의 출발점 = MealPlanSlot

### 완료 조건
- 문서 간 용어와 구조가 모두 일치한다
- 기존 Sprint 이력을 삭제하지 않는다
- 구조 재정의가 기존 계획 대체가 아니라 보강 작업임이 명시된다

## 3. Step 2 — `schema.prisma` 수정

### 목적
문서 기준을 DB 구조에 반영한다.

### 해야 할 일
1. 기존 MealPlan 관련 모델 구조 확인
2. `MealPlanGroup`에서 lineup 책임 제거
3. `MealPlan`에 lineup 관계 배치
4. `MealPlanSlot`에 실행 배정 필드 추가
5. relation / unique / nullable / index 재정리
6. 하위 모델(MealCount, MealPlanAccessory, MaterialRequirement, CookingPlan)에 미치는 영향 확인

### 주의
- 기존 CRUD가 왜 깨질 수 있는지 미리 체크한다
- migration이 너무 많은 모델에 파급되지 않도록 범위를 명확히 한다
- 추후 Sprint 6에서 구현될 Lineup 도메인과 충돌하지 않게 naming을 정리한다

### 완료 조건
- `prisma validate`
- `prisma generate`
- `prisma migrate dev`
- Prisma Studio에서 관계 확인

---

## 4. Step 3 — `seed.ts` 수정

### 목적
새 구조가 실제 데이터로도 동작하는지 검증할 수 있게 한다.

### 해야 할 일
- 날짜 그룹 예시 추가
- 같은 날짜 내 여러 lineup 예시 추가
- MealPlan 카드 예시 추가
- Slot 단위 recipe / supplier item / productionLine 예시 추가
- count 데이터 예시 추가
- accessory 예시 추가
- 재실행 가능하게 유지

### 완료 조건
- seed 재실행 시 충돌이 없다
- UI와 서비스 테스트에 필요한 최소 구조가 있다
- 예제 데이터만 봐도 새 구조를 이해할 수 있다

---

## 5. Step 4 — `meal-plan.schema.ts` 수정

### 목적
새 구조에 맞는 입력 검증 체계를 만든다.

### 해야 할 일
- Group create/update schema 정비
- Plan create/update schema 정비
- Slot create/update schema 정비
- copy schema 정비
- state change schema 정비
- count 관련 schema 분리 준비

### 완료 조건
- Group / Plan / Slot 책임이 schema 수준에서 분리된다
- count와 status가 섞이지 않는다
- action에서 바로 사용할 수 있다

---

## 6. Step 5 — `meal-plan.service.ts` 재정비

### 목적
도메인 규칙을 구조 재정의 기준에 맞게 다시 세운다.

### 해야 할 일
- Group CRUD 재검토
- Plan CRUD 재검토
- Slot CRUD 재검토
- copy 로직 수정
- delete cascade 로직 수정
- status transition 유지
- assignment validation 추가
- include 상수 정비

### 특히 확인할 것
- delete는 어떤 단위까지 cascade인가
- copy 시 어떤 필드를 복제하고 어떤 필드는 초기화하는가
- lineup이 Plan으로 이동했을 때 목록/상세 include가 어떻게 달라지는가
- supplier item / production line 연결이 null 허용인지 필수인지

### 완료 조건
- service 단위 테스트 가능한 구조
- transaction 정리 완료
- 새 schema와 모순 없음

---

## 7. Step 6 — `meal-plan.action.ts` 재정비

### 목적
새 구조를 UI와 연결할 수 있도록 action 계층을 정렬한다.

### 해야 할 일
- create/update/delete/copy/state-change action 입력 갱신
- permission 유지
- audit log 유지
- handleActionError 유지
- actionOk/actionFail 반환 정렬
- 필요한 revalidatePath 재정리

### 완료 조건
- UI가 새 구조에 맞는 action을 호출할 수 있다
- 기존 공통 action 패턴을 깨지 않는다
- error mapping이 유지된다

---

## 8. Step 7 — `/meal-plans/page.tsx` 재구성

### 목적
사용자가 새 구조를 자연스럽게 다룰 수 있게 화면을 재정렬한다.

### 해야 할 일
- 날짜 그룹 목록 유지
- 상세 영역에서 MealPlan 카드 구조를 식사타입 × lineup 기준으로 재정렬
- Slot 상세 편집 도입
- supplier item / productionLine 입력 UI 추가
- 복사 / 상태 변경 / 식단 추가 / 삭제 UX 유지
- lineup 미구현 상태에서는 임시 입력 UX를 유지하되, Sprint 6 이후 교체 가능하게 설계

### 완료 조건
- 한 날짜에 여러 lineup을 표현 가능
- 슬롯 배정이 UI에서 관리 가능
- 기존 toast / dialog / alert 패턴 유지

## 9. Step 8 — 자동생성 연결 기준 정리

### 목적
Sprint 2 후반부의 MealCount / MaterialRequirement / CookingPlan 작업이 흔들리지 않도록 입력 계약을 명시한다.

### 해야 할 일
- MealCount 입력 시점 정의
- MealPlanAccessory 적용 방식 정의
- MaterialRequirement 산출 기준 정의
- CookingPlan 연결 기준 정의
- 자동생성 멱등성 규칙 정의
- 재실행 시 덮어쓰기/누적/재생성 정책 정리

### 완료 조건
- “어떤 입력이 무엇을 만든다”가 문서로 명확하다
- Sprint 2 Phase 8, 9를 바로 이어갈 수 있다
